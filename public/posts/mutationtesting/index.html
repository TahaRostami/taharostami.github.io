    Introduction to Code-based Mutation Testing

    June 5, 2021
    Mutation Testing, Master Thesis
    code-based mutation testing, mutation testing, mutation analysis

    preface #

    A piece of software may have bugs and the presence of bugs may cause problems. So Software Industry uses some methods to reduce the risk of existence of bugs. One of this method is Software Testing which is a process of evaluating a software system to ensure its correct and optimal behavior or to find possible errors in it. One of the most effective techniques in Software Testing is Mutation Testing.

    Testing can apply to different software engineering artifacts, but now we focus on Testing software program codes. In the next sections, we introduce you with code-based mutation testing (i.e. testing software codes using mutation testing) and the concepts related to it.

    Mutation Testing Overview #

    We design an example to explain the mutation testing step by step. Assume we ask a programmer to write a method name Compare with the following behavior:

    \( Compare(x,y)= \begin{cases} \space\space\space1 &\text{if } x>y \\ \space\space\space0 &\text{if } x=y \\ -1 &\text{if } x<y \end{cases} \)

    Now Assume a programmer write a program like below, for this purpose:

    Compare(x, y)
1. const c=1
2. var res=c
3. if x>=y:
4.   res=-1/res
5.   if x==y:
6.     res=res+c
7.     print(res)

    But maybe the programmer made a mistake in writing the program code. So to enusring about bug freeness of the program, we ask a test desginer to design test suite for testing the program.

    Now Assume a test designer desgin the following as the test suite:

    Test input Test Oracle
    Test case 1 (x:0 , y:0) expected std. output: 0
    Test case 2 (x:8 , y:8) expected std. output: 0

    Now we have both source code and test suite for it. So we can execute the program with respect to each test case of the test suite. If program behavior(i.e. it’s output) is expected with respect to each test case, it’s fine. But if the program behave as not expected this means we find a bug in the program.

    If we excute the above program with respect to it’s test suite, the program behave as excpected in all cases. In this case, two cases are possible:

    1. the program is bug free.
    2. the program may have a bug but because of our poor test suite, we did not find it yet!

    The question 2 implies that Are Our test suite is strong enough to reveal potential bugs of the program?

    Mutation Testing is one the approach we can use for answering the question 2, which we will talk about now.
